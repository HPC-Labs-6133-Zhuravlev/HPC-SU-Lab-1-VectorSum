Такой подход позволяет отдельно оценить:
- реальную производительность всей GPU-функции целиком;

## Теоретическая основа

### Суммирование на CPU
На CPU используется обычный последовательный алгоритм:
- создается переменная суммы;
- в цикле перебираются все элементы массива;
- каждый элемент добавляется к общей сумме.

Алгоритм имеет сложность `O(N)` и выполняется одним потоком.

### Суммирование на GPU
На GPU суммирование реализовано через **параллельную редукцию**.

Идея редукции состоит в том, что сумма большого массива может быть представлена как сумма независимых частичных сумм:

$$
(a_1 + a_2 + \dots + a_k) + (a_{k+1} + \dots + a_n)
$$

Это позволяет разбить вычисление между большим числом потоков GPU.

---

## Что именно было распараллелено
В GPU-версии распараллелена операция суммирования элементов вектора.

Каждый поток GPU:
- считывает до двух элементов массива;
- вычисляет локальную частичную сумму;
- записывает промежуточный результат в `shared memory` (разделяемую память).

Далее внутри блока выполняется древовидная редукция:
- на каждом шаге число активных потоков уменьшается в 2 раза;
- частичные суммы объединяются;
- в результате каждый блок формирует одну итоговую частичную сумму.

После этого для массива частичных сумм запускаются следующие проходы редукции, пока не останется одно итоговое значение.

- `Err_full = |CPU_sum - GPU_full_sum|`

Для корректной программы обе ошибки должны быть равны нулю.

## Измерение времени

### 1. CPU time
Измеряется время последовательного суммирования элементов вектора на CPU.

### 2. GPU full time
Измеряется полное время GPU-вычисления:
- копирование массива на устройство;
- запуск одного или нескольких CUDA-ядер редукции;
- копирование результата обратно на хост.

### 3. GPU kernel only time
Измеряется только время CUDA-ядер при помощи `cudaEventRecord()` и `cudaEventElapsedTime()`.

Такой замер позволяет понять, насколько быстро работает само ядро без учета накладных расходов на передачу данных.

---

## Формулы
Полное ускорение GPU относительно CPU:

```text
Speedup_Full = CPU_Time_ms / GPU_Full_Time_ms
```

Ускорение только по времени ядра:

```text
Speedup_Kernel = CPU_Time_ms / GPU_Kernel_Time_ms
```

Если значение больше `1`, значит соответствующий GPU-вариант быстрее CPU.

## Анализ результатов

### 1. Корректность вычислений
Во всех экспериментах выполнены условия:
- `CPU_sum = GPU_full_sum = GPU_kern_sum`;
- `Err_full = 0`;
- `Err_kernel = 0`.

Следовательно, CPU- и GPU-реализации работают корректно.

### 2. Сравнение CPU и GPU full
Полное GPU-время оказалось больше CPU-времени для всех исследованных размеров массива.

Например, при `N = 1 000 000`:
- `CPU_Time_ms = 0.4176` мс;
- `GPU_Full_Time_ms = 0.9971` мс;
- `Speedup_Full = 0.4188`.

Это означает, что при полном учете передачи данных GPU в данной задаче оказался медленнее CPU.

### 3. Сравнение CPU и GPU kernel only
Если учитывать только время выполнения CUDA-ядер, результат меняется:
- начиная с `N = 50000`, значение `Speedup_Kernel > 1`;
- при `N = 1 000 000` получено `Speedup_Kernel = 7.8776`.

Это означает, что само CUDA-ядро работает быстрее CPU, а основная потеря времени связана с копированием данных между CPU и GPU.

### 4. Основной вывод по производительности
Полученные результаты показывают важное различие между двумя сценариями:

1. **End-to-end измерение** (`GPU full`) показывает реальное полное время работы функции на GPU.  
   В этом режиме GPU проигрывает CPU из-за накладных расходов на передачу данных и запуск ядра.

2. **Kernel-only измерение** показывает производительность только вычислительной части.  
   В этом режиме GPU уже демонстрирует ускорение и становится значительно быстрее CPU на больших размерах вектора.

Именно поэтому для оценки CUDA-программ полезно измерять не только полное время, но и чистое время ядра.
